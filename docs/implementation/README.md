public comparison work are deferred until post-MVP when the API and major core
development are stable. The full local RC gate now passes across `fleury`,
`fleury_widgets`, `fleury_git`, and `fleury_example_console`; remaining
completion work is external launch evidence, not more known local core work.
M2.12 public adoption collateral is dispositioned out of the MVP cycle until
API freeze; internal positioning, local distribution planning, and scorecard
scaffolding are enough for the current checkpoint. The
[MVP readiness report](mvp-readiness-report.md), generated by
`dart tool/fleury_dev.dart mvp-readiness`, is the current combined external
evidence gate. `dart tool/fleury_dev.dart mvp-final-gate` is the full final
command: it runs the local RC gate and then enforces external readiness.
`dart tool/fleury_dev.dart mvp-evidence-refresh` regenerates all generated MVP
and post-MVP Windows evidence packets from the current matrix state.
`terminal-matrix-accept`
records human acceptance for explainable `needsAttention` entries so strict
gates can consume reviewed captures without hand-editing JSON.
MVP execution rule: keep existing peer artifacts and benchmark tooling healthy
only when they directly support core/API stabilization. Do not add new peer
fixtures, repeated-run families, full-scale parity runs, cross-machine runs, or
public comparison assets until the launch API and major core implementation are
stable.
**Owner note:** Keep this folder current while implementation proceeds. It is
the working memory for moving Fleury from today's framework to the leading
reactive TUI vision.

## How To Use This Folder

- Start with [milestones.md](milestones.md) before beginning implementation
  work.
- Confirm the active goal and working rules in
  [execution-journal.md](execution-journal.md) before changing code.
- Read the latest entries in [execution-journal.md](execution-journal.md) to
  recover tactical context from prior implementation turns.
- Update the relevant workstream file when implementation reveals new
  constraints, risks, or acceptance evidence.
- Record durable decisions in [decision-log.md](decision-log.md), not only in
  chat or PR notes.
- Record tactical findings, false starts, validation results, and next steps in
  [execution-journal.md](execution-journal.md).
- Link completed tasks to RFCs, tests, benchmarks, examples, PRs, or commit
  SHAs in the `Evidence` field.
- Prefer milestone-level tracking. Do not turn these files into fragile
  file-by-file TODO lists unless a file path is needed to avoid ambiguity.

When starting a task:

1. Mark the task as in progress with a short dated note.
2. Add any local implementation context that would help the next engineer.
3. Link related roadmap, RFC, or workstream sections.
4. Add or update an entry in [execution-journal.md](execution-journal.md) if the
   task spans more than a trivial edit.

When completing a task:

1. Check the task box.
2. Fill in the `Evidence` line with tests, RFCs, examples, benchmarks, or PRs.
3. Add one or two notes about constraints discovered during implementation.
4. Update [decision-log.md](decision-log.md) if the task resolved or reversed
   a design decision.
5. Append validation results and next steps to
   [execution-journal.md](execution-journal.md).

When revising the plan:

1. Preserve old decisions unless they are explicitly reversed.
2. Add a dated note explaining why the plan changed.
3. Keep roadmap-level strategy in the roadmap and execution state in this
   folder.
