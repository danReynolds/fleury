## Why Fleury

<CardGrid>
  <Card title="Flutter's architecture, for cells" icon="puzzle">
    Widgets, elements, render objects, and a semantics tree — incremental
    rebuilds, layout, and paint over a grid of cells instead of pixels.
  </Card>
  <Card title="One app, two surfaces" icon="laptop">
    The same app renders to a real terminal *or* the browser. A parity oracle
    keeps the two surfaces honest.
  </Card>
  <Card title="Built for agents" icon="seti:default">
    A semantics tree — roles, state, actions — ships as a headline feature, so
    tests and AI agents can *drive* the UI, not screen-scrape it.
  </Card>
  <Card title="Runs anywhere" icon="rocket">
    Embed widgets client-side with dart2js (no backend), or stream a full native
    app to the browser with <code>fleury serve</code>.
  </Card>
</CardGrid>

## How it works

`dart2js` compiles the widget tree plus the Fleury framework to JavaScript, and
it renders into a DOM grid of cells right here in the page. The same widget tree
runs natively in a terminal through a POSIX/Windows driver. See
[Core and targets](/architecture/core-and-targets/) for how the pieces fit, and
[Serving and embedding](/architecture/serving-and-embedding/) for the two ways
to put Fleury on the web.
