None of these are screenshots — each is the real Fleury widget tree, compiled to
JavaScript with `dart2js` and running live in your browser: the clock ticks, the
bars stream, and the table responds to your clicks. Tab through them in the title
bar, or watch them cycle. The *same* code runs in a terminal.

And dropping it into *your* page is the fun part: hand Fleury a host element and a
widget tree, and a `dart2js` bundle does the rest — no iframe, no backend, no
screenshot. Every live demo on this site is one line like this:

```dart
mountApp(() => const Dashboard(), into: appHost);
```

## Why Fleury

<CardGrid>
  <FeatureCard icon="puzzle" title="A real widget tree, for cells" href="/architecture/core-and-targets/" cta="How it fits together">
    Widgets, elements, render objects, and a semantics tree — incremental
    rebuilds, layout, and paint over a grid of cells instead of pixels.
  </FeatureCard>
  <FeatureCard icon="laptop" title="One app, two surfaces" href="/comparison/" cta="Why Fleury">
    The same app renders to a real terminal *or* the browser. A parity oracle
    keeps the two surfaces honest.
  </FeatureCard>
  <FeatureCard icon="seti:default" title="Built for agents" href="/architecture/agents-and-semantics/" cta="How agents drive it">
    A semantics tree — roles, state, actions — ships as a headline feature, so
    tests and AI agents can *drive* the UI, not screen-scrape it.
  </FeatureCard>
  <FeatureCard icon="rocket" title="Runs anywhere" href="/architecture/serving-and-embedding/" cta="Serving & embedding">
    Embed widgets client-side with dart2js (no backend), or stream a full native
    app to the browser with <code>fleury serve</code>.
  </FeatureCard>
</CardGrid>

## How it works

*One widget tree, two surfaces* is the headline — underneath is a retained-mode
engine that turns your widgets into cells and paints them wherever you point it:
a terminal, a browser, a streamed session. These pages are the tour.
