Fleury is a retained-mode UI framework — the familiar Flutter model of a widget
tree rebuilt declaratively, `setState` and all — that paints to a **grid of
cells** instead of pixels. One widget tree renders in a terminal *and* a browser.
Fleury can also project that tree into a **semantic graph**, so a test or an
agent can read it like a screen reader.

The status panel below is live — a real Fleury tree compiled to JavaScript and
running in your browser, from the *same* Dart you'll run in a terminal.

<FleuryExample id="home.monitor" cols={34} rows={9} />

In about five minutes you'll build exactly this from scratch, run it in your
terminal, and watch it run in a browser. Let's go.

## 1. Create a project

```sh
dart create my_app
cd my_app
```

Add the framework and the widget library to `pubspec.yaml`:

```yaml
publish_to: none

dependencies:
  fleury:
    git:
      url: https://github.com/danReynolds/fleury.git
      path: packages/fleury
  fleury_widgets:
    git:
      url: https://github.com/danReynolds/fleury.git
      path: packages/fleury_widgets
  fleury_web:
    git:
      url: https://github.com/danReynolds/fleury.git
      path: packages/fleury_web
  web: ^1.1.1

# Until the packages are published together, keep fleury_widgets' framework
# dependency on the same Git checkout as the app's direct dependency.
dependency_overrides:
  fleury:
    git:
      url: https://github.com/danReynolds/fleury.git
      path: packages/fleury
```

Then run `dart pub get`. The temporary override is required only for the Git
install; remove it once you switch both packages to hosted releases.

`fleury` is the framework plus its theme-free primitives (layout, `Text`, text
editing, focus), and `fleury_widgets` is the theme-driven control library on
top (buttons, selects, tables, charts). `fleury_web` and `web` are the browser
host and DOM bindings used in step 5. The [core and
targets](/fleury/architecture/core-and-targets/) page explains the split.

## 2. Hello, cells

A Fleury program is a widget tree handed to `runApp`. `FleuryApp` is the
lightweight app shell: its `home` screen gets app-wide navigation, theming, and
command/status scopes. You build everything beneath it with the familiar
`Widget` / `build` / `setState` model — it just paints to a grid of cells
instead of pixels.

```dart title="bin/my_app.dart"
import 'package:fleury/fleury.dart';

void main() => runApp(
      const FleuryApp(
        title: 'My app',
        home: Center(child: Text('Hello, cells!')),
      ),
    );
```

Run it — `Ctrl-C` quits:

```sh
dart run bin/my_app.dart
```

You should see *Hello, cells!* centered in your terminal.

## 3. Add state

Real apps change over time. Make the app stateful and tick a counter.
`setState` marks the widget dirty; Fleury rebuilds only that path,
re-lays-out, and diffs the new cell grid against the old, so an idle frame costs
nothing.

```dart title="bin/my_app.dart"
import 'dart:async';

import 'package:fleury/fleury.dart';
import 'package:fleury_widgets/fleury_widgets.dart';

void main() => runApp(
  const FleuryApp(title: 'Status monitor', home: StatusApp()),
);

class StatusApp extends StatefulWidget {
  const StatusApp({super.key});

  @override
  State<StatusApp> createState() => _StatusAppState();
}

class _StatusAppState extends State<StatusApp> {
  var _tick = 0;
  Timer? _timer;

  @override
  void initState() {
    super.initState();
    _timer = Timer.periodic(const Duration(seconds: 1), (_) {
      setState(() => _tick++);
    });
  }

  @override
  void dispose() {
    _timer?.cancel();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    return Padding(
      padding: const EdgeInsets.all(1),
      child: Column(
        crossAxisAlignment: CrossAxisAlignment.stretch,
        children: [
          Text('uptime: ${_tick}s'),
          const SizedBox(height: 1),
          ProgressBar(value: (_tick % 60) / 60),
        ],
      ),
    );
  }
}
```

Run it again — the progress bar fills once a minute.

## 4. Compose widgets

The widget library is broad — [browse the catalog](/fleury/widgets/) for API details and
live examples across the main widget families. Charts, meters, and the rest
compose like any other widget. Replace the `build` method with the panel behind
the live example at the top of this page:

```dart title="bin/my_app.dart"
@override
Widget build(BuildContext context) {
  return Padding(
    padding: const EdgeInsets.all(1),
    child: Column(
      crossAxisAlignment: CrossAxisAlignment.stretch,
      mainAxisSize: MainAxisSize.min,
      children: [
        Gauge(value: 0.62, label: 'CPU'),
        Gauge(value: 0.81, label: 'MEM'),
        Gauge(value: 0.34, label: 'DISK'),
        const SizedBox(height: 1),
        Sparkline(data: const [3, 5, 4, 8, 6, 9, 7, 5, 8, 6]),
      ],
    ),
  );
}
```

Those are fixed sample values. To make the panel live, feed them from state and
call `setState` — exactly the way step 3 drove the progress bar from `_tick`. (If
you do drop the timer, remove the now-unused `_tick` and `_timer` too.) State,
lifecycle, composition: the same three moves, however rich the screen gets.

## 5. Run the same widget tree in a browser

The `StatusApp` widget tree above is portable; the host entrypoint and imports
are target-specific. For a client-side embed, move the reusable tree into a
web-safe library that imports `package:fleury/fleury_core.dart` and
`package:fleury_widgets/fleury_widgets_web.dart`.

Move `StatusApp` and `_StatusAppState` into `lib/status_app.dart`,
keeping their bodies unchanged and using these imports at the top:

```dart title="lib/status_app.dart"
import 'dart:async';

import 'package:fleury/fleury_core.dart';
import 'package:fleury_widgets/fleury_widgets_web.dart';
```

The native entrypoint remains small and target-specific:

```dart title="bin/my_app.dart"
import 'package:fleury/fleury.dart';
import 'package:my_app/status_app.dart';

void main() => runApp(
  const FleuryApp(title: 'Status monitor', home: StatusApp()),
);
```

Then mount the shared tree from the browser entrypoint:

```dart title="web/main.dart"
import 'package:fleury/fleury_core.dart';
import 'package:fleury_web/fleury_web.dart';
import 'package:my_app/status_app.dart';
import 'package:web/web.dart' as web;

Future<void> main() async {
  final host = web.document.getElementById('app')!;
  await mountApp(
    () => const FleuryApp(title: 'Status monitor', home: StatusApp()),
    into: host,
  );
}
```

The terminal entrypoint still imports `fleury.dart` and calls `runApp`. You can
also use the local **serve** bridge to preview that native app in a browser tab
without changing its imports. Both modes, the compile command, and when to
choose each are in [Deployment & distribution](/fleury/guides/deployment/).
