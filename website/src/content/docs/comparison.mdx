Fleury brings a real UI framework to the terminal — the retained widget model that
powers Flutter, rebuilt for a grid of cells — and combines it with two
less-common capabilities: a UI that **agents and tests can read and drive**, and
**one app that runs in a terminal _or_ a browser**. The terminal already has
excellent frameworks; this is the particular combination Fleury is built for.

## A closer look

### Flutter's model, rebuilt for cells

Fleury is a retained-mode framework: a **widget** tree, an **element** tree that
holds your state, a **render** tree that lays out and paints, and a **semantics**
tree beside them — the same four-tree architecture that powers Flutter,
specialized for a grid of cells. You get a real layout engine (constraints down,
sizes up; rows, columns, flex), composition, `StatefulWidget` / `setState`, and
hot reload in a running terminal. If you've written Flutter it transfers directly;
if you haven't, it's a small model to pick up.

That foundation is why the widget set runs deep — charts (`Gauge`, `Sparkline`,
`LineChart`, `BarChart`, `Histogram`, `Heatmap`), virtualized data (`DataTable`,
`TreeTable`), document viewers (`CodeView`, `MarkdownView`, `JsonView`,
`DiffView`), and surfaces shaped for agent and dev tooling (`MessageList`,
`PatchReview`, `CommandPalette`, `TraceTimeline`). Eighty-plus widgets in the box,
each composable like any other. Tab through a few — real widgets, running live;
click into one and use it (type, arrow, select):

### A UI built for agents and tests

On agent-reachable and browser surfaces, Fleury maintains a **semantic tree**
alongside the visual tree and updates it when semantics or painted coverage
change. Tests materialize a snapshot when `tester.semantics()` asks, and a plain
terminal run does the same only when a debug consumer asks. Meaningful content
and interactive nodes carry a role plus the labels, values, state, and typed
actions relevant to them, while layout-only structure may be folded away. The
vocabulary is broad and domain-specific: not just `button` and `slider` but
`conversation`, `messageList`, `patchReview`, `commandPalette`, and `traceTimeline`,
with concrete actions like `activate`, `submit`, `select`, `increment`, and `copy`.

Tests assert on *meaning* — "row 3 is selected", "the CPU gauge reads 0.62" — and
survive a re-theme, a layout change, or a port to the browser. An AI agent reads
the tree and drives the UI through those actions instead of scraping ANSI and
guessing keystrokes — point any
[Model Context Protocol](https://modelcontextprotocol.io) host at the app, no
agent-specific integration to write. It's produced by the core, runs alongside the visual frame
without adding terminal-paint work, and crosses the [`fleury serve`](/fleury/architecture/serving-and-embedding/)
wire too — so a served preview is as inspectable as a terminal session. See
[Built for agents](/fleury/architecture/agents-and-semantics/).

### One app, terminal or browser

The same widget tree runs three ways: natively in a real terminal, compiled into a
web page client-side with dart2js (no backend — every demo on this page is exactly
that), or previewed from a native process with [`fleury serve`](/fleury/architecture/serving-and-embedding/).
They aren't separate ports. A **parity oracle** asserts the terminal and the
browser render the same tree, cell for cell, so the two surfaces can't quietly
drift apart — and a `dart:io`-free core is what lets one definition span all three.

### Fast by construction — and measured

A `setState` marks one subtree dirty; only that path rebuilds, re-lays-out, and
re-paints, and the target diffs down to the **cell** — an idle app emits nothing, a
busy one pays only for what moved. A cursor move in a 10,000-character editor costs
about **0.8 ms in the saved development fixture**, tracking the change rather
than how much is on screen. Data widgets are virtualized, so a
hundred-thousand-row table costs about a screenful. The
[benchmark suite](/fleury/architecture/performance/) measures local pipeline work,
real-terminal bytes and process cost, and the browser frame path. The numbers back a
peer-competitive claim across many matched workloads — not a universal "fastest
framework" ranking. The dashboard below
streams on a ticker and repaints only the cells that change:

## Why Dart

The honest first reaction to "a TUI framework in Dart" is *why Dart?* — most
terminal tooling is Rust, Go, or Python. The answer is that Dart is what makes
Fleury's core promise *possible*, not just convenient.

**It compiles to both a native binary and the browser.** Dart AOT-compiles to a
fast, self-contained terminal executable — ship it like a Go binary, no runtime
on the box — *and* compiles to JavaScript with `dart2js`. That dual target is the
whole reason one widget tree can run in a terminal *and* a browser with no second
implementation. In most languages, "the same app, native and on the web" means
two codebases; in Dart it's one source and two `dart compile` invocations.

**The retained UI model is Dart's home turf.** The widget / element / render /
`setState` architecture that makes Fleury a real framework — incremental rebuilds,
a layout engine, composition — was proven at scale in Dart by Flutter, and the
language's ergonomics (const constructors, named parameters, mixins, sound null
safety) co-evolved with exactly this kind of declarative tree. Fleury isn't
bending Dart to fit; it's using it for what it was shaped to do.

**Hot reload, in a terminal.** Save a file and your running TUI updates in
place — widget state, focus, and scroll intact — with a plain `dart run`, in
any editor. This is the Dart VM's stateful code-swapping, the loop Flutter
developers lean on, [built into `runApp`](/fleury/guides/hot-reload/). Peer
stacks iterate by restarting: Textual live-applies CSS edits but restarts for
code changes, and the Go and Rust stacks pair with an external watcher that
reruns the process — either way, your app's state and your place in it are
lost on every edit.

None of this makes Dart the *leanest* option at the floor — Rust boots quicker
and idles smaller, and [the benchmarks](/fleury/architecture/performance/) say so
plainly — or the most common in the space. It makes it the right fit for *this*
job: a real, retained UI framework that spans the terminal and the browser from
one codebase.

Two capabilities carry Fleury's edge — **agents** and **accessibility** both fall
out of a single semantic tree, a pairing no other framework here offers. Hot reload,
browser, and images it leads or ties; components put it in the batteries-included
camp, with the largest built-in set. Where it doesn't win: OpenTUI's terminal 3D,
Rust and Go idle leaner, and — most of all — maturity.

## When to pick something else

Fleury is one good option among several. Reach for a peer when its strengths line
up with yours:

- **nocterm** if you want the same Flutter-in-Dart model in a simpler,
  terminal-focused package, with first-party Riverpod and Bloc integrations.
- **Ratatui** if you're in Rust and want maximum control with an immediate-mode
  model and a mature ecosystem.
- **Textual** if you're in Python and want a batteries-included framework with a
  large widget set, CSS-like styling, and devtools.
- **Bubble Tea** if you're in Go and like the Elm architecture and the Charm
  ecosystem (Bubbles, Lip Gloss, Glamour).
- **Ink** if you're in Node and want to build CLIs with React.

## Fleury's trade-offs

The flip side of that bet: Dart's terminal ecosystem is young — fewer libraries
and fewer examples to crib from than Go, Rust, or Python have. Fleury is also
pre-1.0: the API is still settling, and it's distributed as a Git dependency
today. If you need a long-stable, widely-deployed framework right now, one of the
above may fit better. If the retained model, the batteries-included widgets, the
agent- and accessibility-facing semantics, and the web target are what you're
after, that's exactly the gap Fleury fills.
